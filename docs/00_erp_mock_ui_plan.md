# ERP 웹 UI Mock 데이터 기반 화면 설계 계획서

## 1. 목적

현재 회사는 Delphi로 만든 ERP 프로그램을 사용하고 있으며, 앞으로 웹사이트 또는 웹 기반 ERP 화면을 개발하려고 한다. 이 문서는 실제 데이터베이스 연동 전에 mock 데이터를 사용해 ERP 웹 UI의 화면 구성, 메뉴 구조, 주요 업무 흐름, 우선 제작 범위를 정리하기 위한 계획서이다.

목표는 다음과 같다.

- 기존 Delphi ERP의 업무 흐름을 웹 화면으로 옮길 때 필요한 기본 화면 구조를 잡는다.
- 실제 운영 데이터 없이도 화면을 확인할 수 있도록 mock 데이터 기준을 먼저 정한다.
- 회사 내부 담당자, 개발자, 디자이너가 같은 기준으로 화면을 검토할 수 있게 한다.
- 이후 HTML/CSS/JavaScript 또는 Vue/React/Nuxt 같은 프론트엔드 프로젝트로 바로 확장할 수 있는 구조를 만든다.

## 2. 기본 방향

웹 ERP는 일반 홍보용 웹사이트처럼 화려한 첫 화면보다, 실무자가 매일 반복해서 쓰기 편한 업무 화면이 중요하다. 따라서 화면은 다음 방향으로 설계한다.

- 메뉴는 좌측 사이드바 중심으로 구성한다.
- 상단에는 현재 회사, 사용자, 알림, 검색, 빠른 작업 버튼을 둔다.
- 본문은 목록, 상세, 등록, 수정, 승인, 조회 화면을 빠르게 오갈 수 있게 만든다.
- 정보는 카드 남발보다 표, 필터, 탭, 상태 표시, 요약 지표 중심으로 배치한다.
- 색상은 업무용 프로그램답게 차분하게 사용하되, 상태값은 명확하게 구분한다.
- 모바일보다 데스크톱 업무 환경을 1차 기준으로 삼고, 태블릿까지 대응하는 반응형 구조를 고려한다.

## 3. 예상 사용자

| 사용자 | 주요 목적 | 필요한 화면 |
| --- | --- | --- |
| 경영진 | 매출, 구매, 재고, 생산 현황 확인 | 대시보드, 리포트, 승인 현황 |
| 영업 담당자 | 거래처, 견적, 수주, 매출 관리 | 거래처 관리, 견적서, 주문, 매출 목록 |
| 구매 담당자 | 발주, 입고, 매입 관리 | 구매 요청, 발주서, 입고 현황 |
| 생산 담당자 | 작업 지시, 생산 실적, 공정 관리 | 생산 계획, 작업 지시, 실적 등록 |
| 재고 담당자 | 품목, 창고, 입출고, 재고 수량 관리 | 품목 관리, 재고 현황, 입출고 내역 |
| 회계 담당자 | 매출/매입, 수금/지급, 세금계산서 확인 | 회계 요약, 전표, 거래 내역 |
| 시스템 관리자 | 사용자, 권한, 코드, 환경 설정 관리 | 사용자 관리, 권한 관리, 공통 코드 |

```text
│  └─ 매출 현황
├─ 구매 관리
│  ├─ 구매 요청
│  ├─ 발주 관리
│  └─ 입고 현황
├─ 재고 관리
│  ├─ 품목 관리
│  ├─ 창고 관리
│  ├─ 재고 현황
│  └─ 입출고 내역
├─ 생산 관리
│  ├─ 생산 계획
│  ├─ 작업 지시
│  └─ 생산 실적
├─ 회계 관리

│  ├─ 수금/지급 현황
│  └─ 전표 관리
└─ 시스템 관리
   ├─ 사용자 관리

   └─ 공통 코드 관리
```

## 5. 1차 제작 화면

처음부터 모든 ERP 기능을 만들기보다, ERP 웹 화면의 전체 느낌과 반복 패턴을 확인할 수 있는 대표 화면부터 만든다.

| 우선순위 | 화면 | 목적 |
| --- | --- | --- |
| 1 | 통합 대시보드 | 매출, 구매, 재고, 생산, 승인 현황을 한눈에 확인 |
| 2 | 거래처 관리 | ERP의 기본 마스터 데이터 화면 패턴 확인 |
| 3 | 품목 관리 | 검색, 필터, 상태, 단가, 재고 연결 구조 확인 |
| 4 | 재고 현황 | 창고별/품목별 수량, 안전재고, 부족 상태 표시 |
| 5 | 수주 관리 | 영업 주문의 목록, 상세, 상태 흐름 확인 |
| 6 | 발주 관리 | 구매 발주 목록, 승인 상태, 납기 상태 확인 |
| 7 | 생산 계획 | 작업 일정, 진행률, 담당 라인 표시 |
| 8 | 사용자/권한 관리 | 관리자 화면의 기본 구조 확인 |

## 6. 화면 레이아웃 기준

### 6.1 공통 레이아웃

- 좌측 사이드바: 업무 메뉴, 접기/펼치기, 현재 선택 메뉴 표시
- 상단 헤더: 전역 검색, 알림, 사용자 메뉴, 회사/사업장 선택
- 본문 상단: 화면 제목, 설명 대신 핵심 상태 요약, 주요 액션 버튼
- 필터 영역: 기간, 상태, 담당자, 거래처, 품목 등 업무별 검색 조건
- 목록 영역: 표 형태의 데이터, 정렬, 페이지네이션, 상태 배지
- 상세 영역: 우측 패널 또는 별도 상세 화면으로 구성

### 6.2 데이터 목록 화면

목록 화면은 ERP에서 가장 많이 쓰이는 화면이므로 다음 요소를 기본으로 둔다.

- 검색어 입력
- 기간 선택
- 상태 필터
- 담당자 또는 부서 필터
- 엑셀 다운로드 버튼
- 신규 등록 버튼
- 행 선택 시 상세 정보 표시
- 상태별 색상 배지
- 합계, 건수, 금액 요약

### 6.3 등록/수정 화면

등록/수정 화면은 많은 입력 항목을 다루므로 단순한 긴 폼보다 그룹을 나눠 구성한다.

- 기본 정보
- 거래/금액 정보
- 품목/상세 내역
- 첨부 파일
- 이력 또는 메모
- 저장, 임시저장, 취소 버튼

## 7. Mock 데이터 설계

## 8. 상태값 기준

상태값은 화면에서 빠르게 구분되어야 하므로 이름과 색상 의미를 통일한다.

| 상태 | 의미 | UI 표현 방향 |
| --- | --- | --- |
| 정상 | 문제 없이 진행 중 | 녹색 계열 배지 |
| 대기 | 아직 처리 전 | 회색 또는 파란색 계열 배지 |
| 진행 | 현재 처리 중 | 파란색 계열 배지 |
| 지연 | 납기 또는 처리 기한 초과 | 주황색 계열 배지 |
| 부족 | 재고 또는 자원 부족 | 빨간색 계열 배지 |
| 완료 | 업무 종료 | 어두운 녹색 또는 중립 배지 |
| 취소 | 취소된 업무 | 회색 배지 |

## 9. 디자인 톤

ERP 웹 UI는 다음 분위기를 목표로 한다.

- 실무용, 정돈됨, 빠른 탐색
- 정보 밀도는 높지만 답답하지 않게 구성
- 표와 필터가 중심이 되는 업무형 화면
- 강조 색상은 버튼, 상태, 알림에만 제한적으로 사용
- 대시보드는 숫자 요약과 예외 상황을 빠르게 보여주는 구조
- 불필요한 마케팅 문구나 장식적 이미지는 배제

## 10. 기술 구현 방향

현재 단계에서는 문서와 mock 화면 설계가 우선이며, 다음 단계에서 아래 중 하나로 구현할 수 있다.

| 방식 | 장점 | 사용 시점 |
| --- | --- | --- |
| 정적 HTML/CSS/JavaScript | 빠르게 화면 확인 가능, 실행이 간단함 | 첫 mock UI 제작 |
| Vue 또는 Nuxt | 컴포넌트화, 라우팅, 데이터 바인딩에 유리 | 실제 웹 ERP 전환 검토 |
| React 또는 Next.js | 컴포넌트 생태계가 넓음 | 회사 표준이 React일 경우 |
| Spring Boot + Thymeleaf | 백엔드와 서버 렌더링을 같이 구성 가능 | Java/Spring 기반 ERP 확장 |

1차 prototype은 정적 HTML/CSS/JavaScript로 시작하는 것이 적합하다. 이후 화면 구조가 확정되면 Vue/Nuxt 또는 React/Next로 옮기는 방식이 좋다.

## 11. 작업 단계

### 1단계: 계획서 작성

- ERP 웹 UI의 메뉴 구조 작성
- 우선 제작 화면 선정
- mock 데이터 기준 작성
- 공통 화면 레이아웃 정의

### 2단계: Mock UI 제작

- 정적 웹 화면 기본 구조 생성
- 사이드바, 헤더, 대시보드 구현
- 대표 목록 화면 구현
- mock 데이터를 JavaScript 객체 또는 JSON 형태로 구성

### 3단계: 업무별 화면 확장

- 영업 관리 화면 확장
- 구매/재고 화면 확장
- 생산 관리 화면 확장
- 회계/시스템 관리 화면 확장

### 4단계: 검토 및 개선

- 실제 Delphi ERP 사용자에게 화면 흐름 검토 요청
- 자주 쓰는 버튼과 필터 위치 조정
- 불필요한 항목 제거
- 실제 DB/API 연동 전 필요한 필드 정리

계획서 작성 후 바로 진행할 수 있는 다음 작업은 `index.html`, `css/styles.css`, `js/mock-data.js`, `js/app.js`를 만들어 실제로 브라우저에서 열어볼 수 있는 ERP mock UI를 제작하는 것이다.
