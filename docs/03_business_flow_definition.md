# ERP 웹 UI 업무 흐름 정의서

## 1. 문서 목적

이 문서는 ERP Web Mock UI에서 다루는 주요 업무가 어떤 순서로 연결되는지 정의한다. 화면과 데이터 구조만으로는 실제 ERP 개발 방향이 부족하므로, 수주, 재고, 발주, 생산, 사용자 권한이 어떤 흐름으로 이어지는지 정리한다.

이 문서는 다음 작업에 활용한다.

- 기존 Delphi ERP 업무를 웹 화면 흐름으로 재구성
- 화면 간 이동 기준 정리
- API와 DB 설계 시 업무 상태값 기준 마련
- 실제 사용자 검토 회의에서 업무 누락 여부 확인

## 2. 전체 업무 흐름 요약

현재 mock UI 기준 ERP의 대표 흐름은 다음과 같다.

```text
기준 정보 등록
  ├─ 사용자/권한 등록
  ├─ 거래처 등록
  └─ 품목 등록

영업 흐름
  └─ 수주 등록
      ├─ 재고 확인
      ├─ 생산 계획 생성
      └─ 매출/출고 처리로 확장

구매/재고 흐름
  └─ 재고 부족 확인
      ├─ 발주 생성
      ├─ 입고 처리
      └─ 재고 반영

생산 흐름
  └─ 생산 계획 등록
      ├─ 작업 지시
      ├─ 생산 실적 등록
      └─ 재고 입고 또는 출고로 확장
```

## 3. 기준 정보 흐름

### 3.1 목적

ERP 업무는 거래, 재고, 생산보다 먼저 기준 정보가 준비되어야 한다. 기준 정보가 안정적이지 않으면 수주, 발주, 재고, 생산 데이터가 서로 연결되지 않는다.

### 3.2 주요 기준 정보

| 기준 정보 | 관련 화면 | 설명 |
| --- | --- | --- |
| 사용자 | 사용자 관리 | 로그인, 권한, 담당자 기준 |
| 권한 | 사용자 관리 | 메뉴 접근, 처리 가능 업무 기준 |
| 거래처 | 거래처 관리 | 고객사, 매입처, 협력사 기준 |
| 품목 | 품목 관리 | 제품, 원자재, 부자재 기준 |
| 창고 | 재고 현황 | 재고 보관 위치 기준 |
| 생산 라인 | 생산 계획 | 작업 라인 기준 |

### 3.3 흐름

```text
관리자 로그인
  → 사용자 등록
  → 부서/권한 지정
  → 거래처 등록
  → 품목 등록
  → 창고/라인 기준 연결
  → 업무 데이터 입력 가능
```

### 3.4 검토 포인트

| 질문 | 확인 필요 |
| --- | --- |
| 거래처는 고객사와 매입처를 같이 관리하는가? | 거래처 유형 필요 |
| 품목은 제품/반제품/원자재로 나뉘는가? | 품목 유형 필요 |
| 사용자는 여러 사업장에 접근할 수 있는가? | 사업장 권한 필요 |
| 창고별 접근 권한이 필요한가? | 창고 권한 필요 |

## 4. 영업 수주 흐름

### 4.1 목적

고객 주문을 등록하고, 납기와 생산 가능 여부를 확인한 뒤 생산 또는 출고 흐름으로 연결한다.

### 4.2 관련 화면

| 화면 | 역할 |
| --- | --- |
| 거래처 관리 | 고객사 정보 확인 |
| 품목 관리 | 주문 품목 확인 |
| 수주 관리 | 고객 주문 등록 |
| 재고 현황 | 출고 가능 재고 확인 |
| 생산 계획 | 부족 수량 생산 계획 |
| 대시보드 | 지연 수주, 처리 필요 표시 |

### 4.3 기본 흐름

```text
거래처 확인
  → 품목 확인
  → 수주 등록
  → 납기일 입력
  → 재고 확인
  → 재고 충분: 출고/매출 처리로 이동
  → 재고 부족: 생산 계획 또는 발주 요청
  → 수주 상태 변경
```

### 4.4 상태 흐름

| 상태 | 의미 | 다음 상태 |
| --- | --- | --- |
| 대기 | 수주 접수 후 검토 전 | 진행, 취소 |
| 진행 | 생산/출고 준비 중 | 완료, 지연 |
| 지연 | 납기 초과 또는 생산 지연 | 진행, 완료 |
| 완료 | 납품 또는 매출 처리 완료 | 종료 |
| 취소 | 주문 취소 | 종료 |

Mock 화면에서는 위 상태 전이표를 기준으로 선택 가능한 다음 상태만 표시한다. 저장 로직에서도 같은 규칙을 검증하므로, `완료`나 `취소`처럼 종료된 수주를 다시 `진행`으로 되돌릴 수 없다.

### 4.5 화면 동작 예시

1. 영업 담당자가 `수주 관리`에서 신규 버튼을 클릭한다.
2. 거래처와 품목을 선택한다.
3. 수량, 납기일, 담당자를 입력한다.
4. 시스템이 재고 현황을 확인한다.
5. 재고가 충분하면 출고 가능 상태로 표시한다.
6. 재고가 부족하면 생산 계획 또는 발주 필요 항목으로 표시한다.
7. 대시보드의 처리 필요 영역에 지연 또는 부족 항목이 표시된다.

### 4.6 실제 개발 고려사항

- 수주 헤더와 수주 상세 품목 라인을 분리해야 한다.
- 수주 상태는 생산, 출고, 매출 상태와 연결되어야 한다.
- 납기 지연 판단 기준을 명확히 해야 한다.
- 견적서에서 수주로 전환하는 기능이 필요할 수 있다.

## 5. 구매 발주 흐름

### 5.1 목적

재고 부족, 생산 계획, 구매 요청에 따라 매입처에 발주하고 입고까지 연결한다.

### 5.2 관련 화면

| 화면 | 역할 |
| --- | --- |
| 재고 현황 | 부족 품목 확인 |
| 품목 관리 | 품목 정보와 매입 단가 확인 |
| 거래처 관리 | 매입처 정보 확인 |
| 발주 관리 | 발주 등록과 상태 관리 |
| 대시보드 | 발주 지연, 승인 대기 표시 |

### 5.3 기본 흐름

```text
재고 부족 확인
  → 구매 필요 수량 산정
  → 매입처 선택
  → 발주 등록
  → 승인 요청
  → 발주 진행
  → 입고 처리
  → 재고 반영
```

### 5.4 상태 흐름

| 상태 | 의미 | 다음 상태 |
| --- | --- | --- |
| 대기 | 발주 작성 또는 승인 전 | 진행, 취소 |
| 진행 | 발주 승인 후 입고 대기 | 완료, 지연 |
| 지연 | 납기일 초과 | 진행, 완료 |
| 완료 | 입고 완료 | 종료 |
| 취소 | 발주 취소 | 종료 |

Mock 화면에서는 발주도 수주와 같은 종료 상태 규칙을 적용한다. `완료`로 저장되면 입고 처리가 함께 반영되며, 이후 상태 변경은 막는다.

### 5.5 화면 동작 예시

1. 재고 담당자가 `재고 현황`에서 `부족` 상태 품목을 확인한다.
2. 발주 필요 수량을 검토한다.
3. 구매 담당자가 `발주 관리`에서 신규 발주를 등록한다.
4. 매입처, 품목, 수량, 납기일을 입력한다.
5. 승인 권한자가 발주를 승인한다.
6. 입고 처리 후 재고가 증가한다.

### 5.6 실제 개발 고려사항

- 발주와 입고는 별도 문서로 분리하는 것이 좋다.
- 부분 입고를 지원해야 할 가능성이 높다.
- 구매 단가는 품목 단가 이력과 연결되어야 한다.
- 승인 권한과 승인 이력이 필요하다.

## 6. 재고 관리 흐름

### 6.1 목적

품목별, 창고별 재고를 확인하고 입고, 출고, 조정, 예약 재고를 관리한다.

### 6.2 관련 화면

| 화면 | 역할 |
| --- | --- |
| 품목 관리 | 품목 기준 |
| 재고 현황 | 현재고와 안전재고 확인 |
| 수주 관리 | 출고 예약 |
| 발주 관리 | 입고 예정 |
| 생산 계획 | 생산 투입 또는 생산 완료 입고 |

### 6.3 기본 흐름

```text
품목 등록
  → 창고별 재고 생성
  → 입고 처리
  → 출고 처리
  → 재고 조정
  → 안전재고 비교
  → 부족 품목 알림
```

### 6.4 재고 수량 의미

| 수량 | 설명 |
| --- | --- |
| 현재고 | 장부상 현재 보유 수량 |
| 안전재고 | 최소 확보 기준 수량 |
| 예약재고 | 수주 또는 생산에 이미 배정된 수량 |
| 가용재고 | 실제 사용 가능한 수량 |
| 불량재고 | 품질 문제로 사용 불가한 수량 |

### 6.5 상태 판단 기준

| 조건 | 상태 |
| --- | --- |
| 현재고 > 안전재고 | 정상 |
| 현재고 <= 안전재고 | 부족 |
| 가용재고 <= 0 | 출고 불가 |
| 품목 사용 중지 | 중지 |

### 6.6 실제 개발 고려사항

- 재고 수량은 직접 수정하지 않고 입출고 전표로만 변경해야 한다.
- 로트, 시리얼, 유통기한 관리가 필요한지 확인해야 한다.
- 창고 이동 기능이 필요할 수 있다.
- 실시간 재고와 장부 재고의 차이를 관리해야 한다.

## 7. 생산 관리 흐름

### 7.1 목적

수주 또는 재고 보충 필요에 따라 생산 계획을 만들고, 작업 지시와 생산 실적으로 연결한다.

### 7.2 관련 화면

| 화면 | 역할 |
| --- | --- |
| 수주 관리 | 생산 필요 수량 발생 |
| 품목 관리 | 생산 품목 기준 |
| 재고 현황 | 원자재 재고 확인 |
| 생산 계획 | 생산 일정과 수량 관리 |
| 대시보드 | 생산 지연과 진행률 표시 |

### 7.3 기본 흐름

```text
생산 필요 확인
  → 생산 계획 등록
  → 원자재 재고 확인
  → 작업 라인 배정
  → 작업 지시 생성
  → 생산 실적 등록
  → 완제품 재고 반영
```

### 7.4 상태 흐름

| 상태 | 의미 | 다음 상태 |
| --- | --- | --- |
| 대기 | 계획 등록 후 작업 전 | 진행 |
| 진행 | 생산 작업 중 | 완료, 지연 |
| 지연 | 계획 종료일 초과 | 진행, 완료 |
| 완료 | 생산 실적 처리 완료 | 종료 |

Mock 화면에서는 생산 계획의 상태를 `대기 → 진행 → 완료` 흐름으로 제한한다. `진행`으로 바뀌면 자재 출고가 반영되고, `완료`로 바뀌면 제품 입고와 진행률 100%가 반영된다. `지연` 상태는 생산 중 예외 상태로만 사용하며, 이후 `진행` 또는 `완료`로만 이동할 수 있다.

### 7.5 실제 개발 고려사항

- 생산 계획과 작업 지시를 분리할지 결정해야 한다.
- BOM 기준 자재 소요량 계산이 필요할 수 있다.
- 실적 등록 시 양품, 불량, 작업 시간, 작업자를 기록해야 한다.
- 생산 완료 후 완제품 재고 반영 규칙이 필요하다.

## 8. 사용자/권한 흐름

### 8.1 목적

ERP 사용자별로 메뉴 접근, 데이터 조회, 등록/수정/삭제/승인 권한을 제어한다.

### 8.2 기본 흐름

```text
사용자 등록
  → 부서 지정
  → 역할 지정
  → 권한 그룹 연결
  → 메뉴 접근 가능
  → 업무 처리 이력 기록
```

### 8.3 권한 구분 예시

| 권한 | 설명 |
| --- | --- |
| 조회 | 목록과 상세 조회 가능 |
| 등록 | 신규 데이터 등록 가능 |
| 수정 | 기존 데이터 수정 가능 |
| 삭제 | 데이터 삭제 또는 취소 가능 |
| 승인 | 발주, 수주, 생산 등 승인 가능 |
| 관리자 | 사용자와 공통 코드 관리 가능 |

### 8.4 실제 개발 고려사항

- 메뉴 권한과 데이터 권한을 분리하는 것이 좋다.
- 사업장, 부서, 창고 단위 접근 제한이 필요할 수 있다.
- 처리 이력에는 사용자 ID, 처리 시간, 변경 전/후 값이 남아야 한다.

## 9. 대시보드 흐름

### 9.1 목적

각 업무 흐름에서 발생한 예외와 요약 데이터를 모아 보여준다.

### 9.2 표시 기준

| 대시보드 항목 | 원천 데이터 |
| --- | --- |
| 수주 금액 | 수주 데이터 |
| 재고 경고 | 재고 데이터 |
| 생산 진행 | 생산 계획 데이터 |
| 승인 대기 | 발주, 수주, 생산 승인 데이터 |
| 처리 필요 | 지연, 부족, 대기 상태 데이터 |

### 9.3 실제 개발 고려사항

- 대시보드는 원천 데이터를 직접 입력하지 않는다.
- 각 업무 데이터에서 집계한 결과를 표시한다.
- 사용자 권한에 따라 보이는 지표가 달라질 수 있다.
