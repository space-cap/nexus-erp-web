# ERP 웹 UI 컴포넌트 가이드

## 1. 문서 목적

이 문서는 ERP Web Mock UI에서 반복적으로 사용하는 UI 요소의 기준을 정리한다. 실제 웹 ERP 개발 시 화면마다 다른 방식으로 버튼, 표, 필터, 상태값을 만들지 않도록 공통 컴포넌트 기준을 제공한다.

이 문서는 다음 작업에 활용한다.

- 정적 mock UI를 Vue, React, Nuxt 등으로 컴포넌트화
- 화면별 UI 일관성 유지
- 개발자와 디자이너 간 UI 기준 공유
- 신규 화면 추가 시 재사용 기준 마련

## 2. 전체 디자인 원칙

| 원칙 | 설명 |
| --- | --- |
| 업무 우선 | 장식보다 빠른 조회와 입력을 우선한다 |
| 정보 밀도 유지 | ERP 특성상 많은 데이터를 한 화면에 보여준다 |
| 반복 패턴 통일 | 목록, 필터, 상세, 모달 구조를 통일한다 |
| 상태 명확화 | 정상, 진행, 지연, 부족 상태를 색상과 텍스트로 구분한다 |
| 데스크톱 우선 | 사무실 업무 환경을 기준으로 설계한다 |
| 반응형 보조 | 태블릿, 작은 노트북에서도 깨지지 않게 한다 |

## 3. 레이아웃 컴포넌트

## 3.1 App Shell

### 역할

전체 ERP 화면의 기본 골격이다.

### 구성

| 요소 | 설명 |
| --- | --- |
| 좌측 사이드바 | 업무 메뉴 |
| 상단 헤더 | 전역 검색, 알림, 사용자 |
| 본문 영역 | 대시보드 또는 업무 화면 |

### 사용 기준

- 모든 ERP 화면은 같은 App Shell 안에서 동작한다.
- 메뉴 이동 시 전체 페이지가 바뀌더라도 좌측 메뉴와 상단 헤더는 유지한다.
- 모바일 또는 좁은 화면에서는 사이드바를 접을 수 있게 한다.

## 3.2 Sidebar

### 역할

ERP 업무 메뉴 이동을 담당한다.

### 현재 메뉴

- 대시보드
- 거래처 관리
- 품목 관리
- 재고 현황
- 수주 관리
- 발주 관리
- 생산 계획
- 사용자 관리

### 사용 기준

- 현재 선택된 메뉴는 시각적으로 강조한다.
- 메뉴명은 업무자가 바로 이해할 수 있는 이름을 사용한다.
- 2단계 메뉴가 늘어날 경우 업무 그룹별로 묶는다.

## 3.3 Topbar

### 역할

화면 전체에서 사용하는 검색, 알림, 사용자 정보를 제공한다.

### 구성

| 요소 | 설명 |
| --- | --- |
| 메뉴 버튼 | 좁은 화면에서 사이드바 열기 |
| 전역 검색 | 거래처, 품목, 문서번호 검색 |
| 알림 버튼 | 처리 필요 알림 |
| 사용자 버튼 | 사용자 정보와 계정 메뉴 |

### 사용 기준

- 전역 검색은 현재 화면 검색과 구분해야 한다.
- 실제 개발 시 전역 검색은 통합 검색 API와 연결한다.
- 사용자 버튼에는 로그아웃, 내 정보, 비밀번호 변경을 연결할 수 있다.

## 4. 버튼 컴포넌트

## 4.1 Primary Button

### 역할

가장 중요한 실행 액션에 사용한다.

### 현재 사용 예

- 신규
- 저장
- 처리

### 사용 기준

| 상황 | 사용 여부 |
| --- | --- |
| 신규 등록 | 사용 |
| 저장 | 사용 |
| 승인/처리 | 사용 |
| 단순 조회 | 사용하지 않음 |

## 4.2 Secondary Button

### 역할

보조 액션에 사용한다.

### 현재 사용 예

- CSV
- 취소
- 수정

### 사용 기준

- Primary Button보다 시각적 강조를 낮춘다.
- 취소, 다운로드, 수정, 닫기 같은 보조 작업에 사용한다.

## 4.3 Icon Button

### 역할

공간을 적게 차지해야 하는 도구성 액션에 사용한다.

### 현재 사용 예

- 메뉴 열기
- 알림
- 새로고침
- 모달 닫기

### 사용 기준

- 아이콘만 있을 때는 접근성 라벨이 필요하다.
- 자주 쓰는 기능에만 사용한다.
- 의미가 애매한 아이콘은 텍스트 버튼을 사용한다.

## 5. 검색/필터 컴포넌트

## 5.1 Global Search

### 역할

ERP 전체에서 거래처, 품목, 문서번호 등을 검색한다.

### 실제 개발 고려사항

- 통합 검색 결과 화면이 필요할 수 있다.
- 권한이 없는 데이터는 검색 결과에 나오면 안 된다.
- 검색어 자동완성 기능을 추가할 수 있다.

## 5.2 Module Search

### 역할

현재 업무 화면의 목록 안에서 검색한다.

### 사용 화면

- 거래처 관리
- 품목 관리
- 재고 현황
- 수주 관리
- 발주 관리
- 생산 계획
- 사용자 관리

### 사용 기준

- 목록 상단 필터 영역에 배치한다.
- 검색 대상 필드는 화면마다 다를 수 있다.
- 실제 개발 시 서버 검색과 클라이언트 검색 중 하나를 선택해야 한다.

## 5.3 Status Filter

### 역할

상태값별로 데이터를 조회한다.

### 사용 기준

- 상태 옵션은 현재 화면에서 사용하는 상태만 표시한다.
- `전체` 옵션을 기본값으로 둔다.
- 상태값은 공통 코드와 연결하는 것이 좋다.

## 5.4 Period Filter

### 역할

기간 조건으로 데이터를 조회한다.

### 현재 옵션

- 전체
- 최근 7일
- 이번 달

### 실제 개발 고려사항

- 업무 화면별 기준 날짜가 다르다.
- 수주는 수주일/납기일, 발주는 발주일/납기일, 재고는 처리일 기준이 될 수 있다.
- 실제 개발 시 시작일/종료일 직접 선택 기능이 필요하다.

## 6. 상태 배지 컴포넌트

## 6.1 역할

업무 상태를 빠르게 구분할 수 있도록 텍스트와 색상을 함께 표시한다.

## 6.2 상태별 기준

| 상태 | 의미 | 색상 방향 |
| --- | --- | --- |
| 정상 | 문제 없음 | 녹색 |
| 진행 | 업무 진행 중 | 파란색 |
| 대기 | 처리 전 | 연한 파란색 |
| 지연 | 납기 또는 일정 초과 | 주황색 |
| 부족 | 재고 부족 | 빨간색 |
| 완료 | 업무 종료 | 녹색 |
| 중지 | 사용 중지 | 회색 |

## 6.3 사용 기준

- 상태는 색상만으로 판단하게 만들지 않는다.
- 반드시 텍스트를 함께 표시한다.
- 같은 상태값은 모든 화면에서 같은 색상으로 표시한다.

## 7. KPI 카드 컴포넌트

## 7.1 역할

대시보드에서 핵심 지표를 요약 표시한다.

### 현재 사용 예

- 수주 금액
- 재고 경고
- 생산 진행
- 승인 대기

### 구성

| 요소 | 설명 |
| --- | --- |
| 라벨 | 지표 이름 |
| 값 | 금액, 건수, 라인 수 |
| 설명 | 지표 기준 |
| 아이콘 | 업무 구분 |

### 사용 기준

- KPI는 클릭 시 상세 목록으로 이동할 수 있어야 한다.
- 값은 실제 원천 데이터에서 집계해야 한다.
- 너무 많은 KPI를 첫 화면에 배치하지 않는다.

## 8. Table 컴포넌트

## 8.1 역할

ERP 업무 데이터 목록을 표시한다.

### 현재 기능

- 컬럼 헤더
- 데이터 행
- 숫자 우측 정렬
- 상태 배지
- 선택 행 강조
- 가로 스크롤

### 사용 기준

- 업무 목록은 기본적으로 표 형태를 사용한다.
- 금액, 수량, 진행률은 정렬 기준을 명확히 한다.
- 행 선택 시 상세 패널이 갱신되어야 한다.
- 실제 개발 시 정렬, 페이지네이션, 컬럼 고정 기능을 고려한다.

## 8.2 컬럼 기준

| 데이터 유형 | 표시 기준 |
| --- | --- |
| 코드 | 왼쪽 정렬 |
| 이름 | 왼쪽 정렬 |
| 날짜 | `YYYY-MM-DD` |
| 금액 | 원화 형식, 우측 정렬 |
| 수량 | 천 단위 구분, 우측 정렬 |
| 상태 | 상태 배지 |
| 진행률 | 프로그레스 바 |

## 9. Detail Panel 컴포넌트

## 9.1 역할

목록에서 선택한 행의 상세 정보를 표시한다.

### 구성

| 요소 | 설명 |
| --- | --- |
| 제목 | 거래처명, 문서 번호, 품목명 등 |
| 상태 배지 | 현재 상태 |
| 상세 필드 | 선택 데이터의 주요 값 |
| 액션 버튼 | 수정, 처리 |

### 사용 기준

- 목록과 상세를 같은 화면에서 볼 수 있게 한다.
- 상세 필드는 업무별로 중요도 순서에 맞게 정렬한다.
- 수정과 처리는 권한이 있는 사용자에게만 노출해야 한다.

## 10. Modal 컴포넌트

## 10.1 역할

간단한 신규 등록 또는 확인 작업에 사용한다.

### 현재 사용 예

- 신규 데이터 mock 등록

### 사용 기준

- 짧은 입력 작업에만 사용한다.
- 입력 항목이 많으면 별도 등록 화면으로 분리한다.
- 저장, 취소, 닫기 버튼은 항상 명확해야 한다.

## 11. Toast 컴포넌트

## 11.1 역할

사용자의 액션 결과를 짧게 알려준다.

### 현재 사용 예

- mock 데이터 저장 안내
- 수정/처리 버튼 클릭 안내
- CSV 생성 안내

### 사용 기준

- 성공, 실패, 경고 메시지를 짧게 표시한다.
- 중요한 오류는 토스트만으로 끝내지 않고 화면 내 메시지로도 표시한다.

## 12. 실제 컴포넌트 분리 제안

Vue 또는 React로 전환할 경우 다음 단위로 컴포넌트를 나누는 것이 좋다.

```text
components/
├─ layout/
│  ├─ AppShell
│  ├─ Sidebar
│  └─ Topbar
├─ common/
│  ├─ Button
│  ├─ IconButton
│  ├─ StatusBadge
│  ├─ Toast
│  └─ Modal
├─ data/
│  ├─ FilterBar
│  ├─ DataTable
│  ├─ DetailPanel
│  └─ SummaryStrip
└─ dashboard/
   ├─ KpiCard
   ├─ BarChart
   ├─ TaskList
   └─ TimelineList
```
