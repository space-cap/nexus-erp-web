# ERP 웹 UI 실제 개발 전환 계획서

## 1. 문서 목적

이 문서는 현재 정적 ERP Web Mock UI를 실제 운영 가능한 웹 ERP로 전환하기 위한 개발 계획을 정리한다. 현재 mock UI는 화면 검토용이므로, 실제 개발 단계에서는 인증, 권한, API, DB, 배포, 기존 Delphi ERP와의 연계 방식을 결정해야 한다.

## 2. 현재 상태

| 구분 | 현재 상태 |
| --- | --- |
| 화면 | 정적 HTML/CSS/JavaScript |
| 데이터 | `js/mock-data.js` mock 데이터 |
| 서버 | 없음 |
| DB | 없음 |
| 로그인 | 없음 |
| 권한 | 화면 표시만 있음 |
| 배포 | 로컬 브라우저 실행 |

## 3. 목표 상태

| 구분 | 목표 |
| --- | --- |
| 화면 | Nuxt 기반 웹 ERP 업무 화면 |
| 데이터 | 1차는 mock 데이터 유지, 이후 실제 DB 또는 기존 ERP 데이터 연동 |
| 서버 | 아직 미정, 이후 API 서버 구성 |
| 로그인 | 1차는 UI/mock 권한, 이후 사용자 인증 |
| 권한 | 1차는 mock 권한, 이후 메뉴/업무/데이터 권한 관리 |
| 배포 | 사내 서버 또는 클라우드 배포 |
| 운영 | 백업, 로그, 장애 대응 가능 |

## 4. 개발 전환 방식 선택지

## 4.1 정적 Mock UI 유지 후 단계적 확장

### 설명

현재 HTML/CSS/JavaScript 구조를 유지하면서 화면 검토와 업무 정의를 먼저 충분히 진행한다.

### 장점

- 빠르게 수정 가능
- 비개발자도 브라우저에서 확인 가능
- 실제 개발 전 화면 합의에 유리

### 단점

- 실제 데이터 연동 구조와는 차이가 있다.
- 화면이 많아질수록 코드 관리가 어려워진다.

### 적합한 시점

- 아직 업무 범위가 확정되지 않았을 때
- Delphi ERP 사용자의 의견을 먼저 받고 싶을 때

## 4.2 Nuxt 기반 프론트엔드 전환

### 설명

현재 mock UI를 Nuxt 컴포넌트 기반으로 전환한다. 백엔드는 아직 정하지 않고, 데이터는 Nuxt 프로젝트 내부 mock 데이터를 사용한다.

### 장점

- 컴포넌트 재사용이 쉽다.
- `pages` 기반 라우팅으로 ERP 메뉴 구조를 관리하기 좋다.
- ERP처럼 화면이 많은 시스템에 적합하다.
- 백엔드가 없어도 mock 데이터로 화면 개발을 계속할 수 있다.

### 단점

- 프론트엔드 프로젝트 구조를 새로 잡아야 한다.
- 개발자가 Nuxt와 Vue 컴포넌트 구조에 익숙해야 한다.
- mock 데이터와 실제 API를 교체할 수 있도록 데이터 접근 계층을 미리 분리해야 한다.

### 적합한 시점

- 실제 웹 ERP 프론트엔드 개발을 시작할 때
- 화면이 10개 이상으로 늘어날 때
- 백엔드가 아직 없어도 화면 개발을 먼저 진행하고 싶을 때

## 4.3 Spring Boot + Thymeleaf 방식

### 설명

Spring Boot 서버에서 HTML 화면을 렌더링하는 방식이다.

### 장점

- 백엔드와 화면을 한 프로젝트에서 관리할 수 있다.
- Java/Spring 중심 개발 조직에 적합하다.
- 초기 개발 구조가 단순하다.

### 단점

- 화면 상호작용이 많아지면 프론트엔드 코드가 복잡해질 수 있다.
- SPA 방식보다 화면 전환 경험이 제한적일 수 있다.

### 적합한 시점

- 회사가 Java/Spring 기반을 선호할 때
- 화면보다 서버 업무 로직이 더 중요할 때

## 4.4 기존 Delphi ERP와 병행

### 설명

기존 Delphi ERP를 바로 대체하지 않고, 일부 조회 또는 신규 업무만 웹으로 먼저 만든다.

### 장점

- 리스크가 낮다.
- 기존 ERP 사용을 유지하면서 웹 전환을 검증할 수 있다.
- 업무별로 단계적 전환이 가능하다.

### 단점

- 기존 DB 구조와 웹 API 구조를 맞추는 작업이 필요하다.
- 같은 업무가 Delphi와 웹에 동시에 존재할 수 있다.

### 적합한 시점

- 기존 ERP가 운영 중이고 즉시 교체가 어려울 때
- 조회, 대시보드, 모바일 승인 같은 일부 기능부터 웹으로 만들고 싶을 때

## 5. 추천 개발 방향

현재 상황에서는 다음 순서가 적합하다.

```text
1. 정적 mock UI로 화면과 업무 흐름 합의
2. Nuxt 기반 mock 프론트엔드로 전환
3. 기존 Delphi ERP의 DB 구조와 업무 흐름 조사
4. 웹 전환 대상 업무 선정
5. 백엔드 기술과 API 방식 결정
6. Nuxt의 mock 데이터 접근 계층을 실제 API로 교체
7. 일부 업무부터 실제 데이터 연동
8. 사용자 검증 후 점진적 확장
```

처음부터 전체 ERP를 웹으로 다시 만드는 것보다, Nuxt에서 mock 데이터 기반 화면을 먼저 안정화하고 조회와 대시보드, 거래처/품목 같은 기준 정보 화면부터 실제 데이터로 전환하는 것이 안전하다.

## 6. 1차 실제 개발 범위 제안

## 6.1 포함 기능

| 우선순위 | 기능 | 이유 |
| --- | --- | --- |
| 1 | 로그인 | 모든 업무의 시작점 |
| 2 | 사용자/권한 관리 | 메뉴 접근 제어 필요 |
| 3 | 거래처 조회/상세 | 공통 기준 데이터 |
| 4 | 품목 조회/상세 | 재고/수주/발주/생산 공통 기준 |
| 5 | 재고 현황 조회 | 업무 효과가 바로 보이는 화면 |
| 6 | 대시보드 | 경영진과 관리자에게 유용 |

## 6.2 제외 기능

| 제외 기능 | 이유 |
| --- | --- |
| 전체 회계 처리 | 업무 복잡도가 높음 |
| 생산 실적 전체 등록 | 현장 프로세스 확인 필요 |
| 자동 재고 차감 | DB 구조와 업무 규칙 확인 필요 |
| 복잡한 승인 워크플로우 | 권한 정책 확정 후 진행 |

## 7. 기술 스택 후보

## 7.1 프론트엔드

| 후보 | 장점 | 비고 |
| --- | --- | --- |
| Nuxt 3 | 라우팅, 구조화, 컴포넌트 기반 ERP 화면에 적합 | 1차 추천 |
| Vue 3 | 배우기 쉽고 업무 화면 구성에 적합 | Nuxt 없이 SPA로 갈 때 후보 |
| React | 생태계가 넓음 | 회사 표준이면 선택 |
| 정적 HTML/JS | 빠른 mock 확인 | prototype 전용 |

## 7.2 백엔드

| 후보 | 장점 | 비고 |
| --- | --- | --- |
| Spring Boot | ERP 업무 API에 적합, Java 생태계 안정적 | 추천 |
| Node.js | 프론트엔드와 언어 통일 가능 | 소규모 API에 적합 |
| .NET | Windows/사내 시스템과 친화적 | 회사 기술 스택에 따라 선택 |

## 7.3 DB

| 후보 | 장점 | 비고 |
| --- | --- | --- |
| MS SQL Server | 기존 ERP/Windows 환경과 잘 맞음 | Delphi ERP가 MSSQL이면 우선 |
| PostgreSQL | 안정적이고 비용 부담 적음 | 신규 시스템에 적합 |
| MySQL/MariaDB | 익숙한 개발자가 많음 | 일반 업무 시스템에 적합 |

## 8. API 설계 방향

## 8.1 기본 원칙

| 원칙 | 설명 |
| --- | --- |
| 화면 단위 API보다 업무 단위 API | UI가 바뀌어도 API 재사용 가능 |
| 목록/상세/등록/수정 분리 | ERP 화면 패턴과 잘 맞음 |
| 상태값 코드화 | 화면 표시명과 DB 값을 분리 |
| 페이지네이션 기본 적용 | ERP 데이터는 금방 많아짐 |
| 권한 체크 필수 | 조회와 처리 모두 서버에서 검증 |

## 11. 개발 단계 계획

## 11.1 1단계: 업무 분석

- Delphi ERP 주요 화면 캡처
- 실제 사용자 인터뷰
- DB 테이블 구조 확인
- 자주 쓰는 업무 우선순위 정리

## 11.2 2단계: prototype 고도화

- 현재 mock UI에 실제 필드명 반영
- 거래처/품목/재고 화면 상세화
- 등록/수정 폼 분리
- 화면별 검토 체크리스트 작성

## 11.3 3단계: 기술 검증

- 프론트엔드 프로젝트 생성
- API 서버 기본 구조 생성
- 로그인 prototype
- DB 연결 테스트
- 기존 ERP DB 조회 테스트

## 11.4 4단계: 1차 업무 개발

- 사용자/권한
- 거래처 조회/상세
- 품목 조회/상세
- 재고 현황 조회
- 대시보드 요약

## 11.5 5단계: 사용자 검증

- 실제 사용자 테스트
- 기존 Delphi 화면과 비교
- 누락 필드 보완
- 버튼 위치와 검색 조건 조정

## 11.6 6단계: 운영 준비

- 서버 배포
- 백업 정책
- 로그 관리
- 권한 점검
- 장애 대응 절차

## 12. 리스크와 대응

| 리스크 | 설명 | 대응 |
| --- | --- | --- |
| 기존 업무 복잡도 | Delphi ERP에 숨은 업무 규칙이 많을 수 있음 | 화면 캡처와 사용자 인터뷰 필수 |
| DB 구조 불명확 | 오래된 ERP는 테이블명이 직관적이지 않을 수 있음 | 조회 쿼리와 실제 화면 비교 |
| 전체 재개발 부담 | 모든 기능을 한 번에 만들면 실패 가능성 증가 | 조회/기준 정보부터 단계적 전환 |
| 권한 누락 | 웹에서는 접근 통제가 더 중요함 | 서버 권한 체크 필수 |
| 데이터 정합성 | Delphi와 웹이 동시에 DB를 수정하면 충돌 가능 | 초기에는 조회 중심으로 시작 |

## 14. 다음 작업 제안

이 문서까지 작성되면 웹 ERP 전환의 1차 기획 문서 세트는 갖춰진다. 다음 작업은 두 방향 중 하나로 진행할 수 있다.

| 방향 | 설명 |
| --- | --- |
| 문서 고도화 | API 명세서, DB 테이블 정의서, 화면별 등록/수정 폼 정의서 작성 |
| UI 고도화 | mock UI에 실제 등록/수정 화면, 화면별 상세 탭, 더 현실적인 ERP 필드 추가 |

기존 Delphi ERP의 실제 화면이나 DB 필드 정보를 확보하면 현재 문서와 mock UI를 회사 업무에 더 가깝게 맞출 수 있다.
